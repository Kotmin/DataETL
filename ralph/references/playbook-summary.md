# Ralph Playbook — Key Patterns Reference

Source: https://claytonfarr.github.io/ralph-playbook/ (G. Huntley / Clayton Farr)

## Core Philosophy

- **Fresh context per iteration**: Each Claude invocation starts clean (~40-60% context utilization = "smart zone")
- **File-based state**: Persistent state lives in markdown/json files, not chat history
- **Backpressure**: Tests/builds/lints act as downstream validators that force correctness
- **Operators outside the loop**: Humans observe, tune reactively — don't prescribe per-task
- **Eventual consistency**: Trust iterative self-correction over upfront perfection

## File Architecture

```
PROMPT_build.md    Full agent instructions (phases, guardrails)
PROMPT_plan.md     Planning mode instructions
AGENTS.md          ~60-line operational guide; updated by agents with learnings
                   NOT a changelog. Contains: commands, known failures, conventions
IMPLEMENTATION_PLAN.md  Disposable working task list; LLM-generated, frequently updated
specs/*.md         Source-of-truth requirements (one per topic/concern)
                   Describe WHAT (behavioral outcomes), never HOW (implementation)
```

## Two-Mode Loop

| Mode | Input | Output |
|------|-------|--------|
| PLANNING | specs/*.md or PRD | IMPLEMENTATION_PLAN.md |
| BUILDING | IMPLEMENTATION_PLAN.md | Code + passing tests + updated plan |

## AGENTS.md Rules

- ~60 lines max
- Contains: build/run/test commands, validation steps, operational learnings
- Agents MAY update Learnings section
- NOT a progress tracker or changelog
- Is the "loopback" mechanism — specifies what "run tests" means concretely

## IMPLEMENTATION_PLAN.md Rules

- Bullet-point task list (format determined by LLM)
- Generated by planning mode, updated by building mode
- Disposable — regenerate when wrong, stale, or confusing
- Treat regeneration as cheap, not a failure

## Subagent Strategy

- **Reads/searches**: Parallel subagents (up to many)
- **Build/tests**: Single subagent (intentional bottleneck = backpressure)
- **Complex reasoning**: More capable models when needed
- Main agent = coordinator/scheduler, not implementer

## Prompt Language Patterns

- "study" (not "read") — implies analysis, not just scanning
- "don't assume not implemented" — always verify before building
- "using parallel subagents" — for information gathering
- "only 1 subagent for build/tests" — enforces backpressure
- Guardrail numbering: 999+, 9999+, 99999+ (higher = more critical)

## Specs Format

- One file per topic of concern
- Topic scope test: "Can you describe this in one sentence without 'and'?"
- Acceptance criteria: observable behavioral outcomes only
- Never prescribe HOW — only WHAT to verify

## Backpressure Beyond Code

- Programmatic: tests, typecheck, lint (measurable)
- LLM-as-Judge: for subjective criteria (tone, UX) — binary pass/fail fixture

## Safety Model

- `--dangerously-skip-permissions` required for autonomous operation
- Security via sandbox isolation (Docker, network restrictions)
- Escape hatches: Ctrl+C stops loop, `git reset --hard` reverts

## Steering Philosophy

1. Start with minimal AGENTS.md
2. Spot-test desired behaviors
3. Watch initial loops, identify failure patterns
4. Tune **only as needed**: prompt guardrails, AGENTS.md updates, codebase utilities
5. Signs aren't just prompts — code patterns, tests, utilities are also signals

## Work-Scoped Branches

Create scoped IMPLEMENTATION_PLAN.md upfront per branch:
```bash
./loop.sh plan-work "user authentication with OAuth"
```
Ralph builds from already-focused list without semantic filtering at runtime.

## Plan Regeneration Triggers

- Ralph going off-track / wrong implementations
- Plan stale or doesn't match code state
- Too much clutter from completed items
- Significant spec changes
- Confusion about actual completion state
